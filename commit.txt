feat(nip-0004): Phase 4 — Mailbox Primitive

Implements the first stateful Protocol Object type per NIP-0004 §10.
Mailbox introduces queue + mutation semantics on top of the Phase 1
Protocol Object Registry (0xFF01) and Phase 2 Deferred Queue (0xFF02),
making it the canonical reference primitive for all subsequent stateful
PO types (Session, ContentRef, Subscription, GameRoom).

== New precompiles ==

0x2C novaMailboxManager — lifecycle:
  0x01 createMailbox(capacity, retention, postage, aclMode, expiry,
                     rentPrepay, aclCount, ACL[]) → 32-byte mailbox ID
  0x02 configureMailbox(id, ...same fields)         WRITE, owner-only
  0x03 destroyMailbox(id)                           WRITE, owner-only
  0x04 getMailboxConfig(id)                         READ

0x35 novaMailboxOps — messaging hot path:
  0x01 sendMessage(mailboxID, payloadHash, postage) WRITE, deferred
  0x02 recvMessage(mailboxID)                       WRITE, owner-only
  0x03 peekMessage(mailboxID)                       READ
  0x04 countMessages(mailboxID)                     READ

Two-precompile design (instead of folding ops into 0x2C) keeps the
gas table compact and lets the ops surface evolve (batch recv, future
opcode bridge MSEND/MRECV/MPEEK/MCOUNT in Phase 12) without churning
the manager.

== Storage layout ==

Mailbox PO body lives at 0xFF01 (Phase 1 Registry). StateData carries
RLP-encoded MailboxConfig (capacity, retention policy/blocks, ACL mode
+ entries, min postage). Owner, expiry, last-touched, rent balance use
the canonical PO fields.

Per-mailbox queue + counters live at NEW system address 0xFF04
(MailboxOpsAddr), distinct from 0xFF01/0xFF02/0xFF03 — each subsystem
owns its reserved account so corruption in one cannot cross-contaminate
another. Storage keys:

  keccak256("mb_head", id)         → uint64 next msg index to recv
  keccak256("mb_tail", id)         → uint64 next msg index to write
  keccak256("mb_count", id)        → uint64 live message count
  keccak256("mb_pending", id)      → uint64 in-flight (sent but not
                                     yet delivered)
  keccak256("mb_msg_marker"|len|chunks|chunk, id, idx) → chunked RLP
                                     of MailboxMessage
  keccak256("mb_owner_count"|slots|idx|slot_of, owner) → mailbox-
                                     specific owner index (in addition
                                     to the Phase 1 generic PO index)

Dual owner-index update keeps getProtocolObjectsByOwner working AND
gives type-filtered getMailboxByOwner without a full PO scan.

== Deferred delivery (block N → N+1) ==

sendMessage at block N:
  1. Lookup mailbox PO; reject if absent or expired.
  2. Decode MailboxConfig; check ACL (Open / Whitelist / Blacklist;
     owner-to-self always allowed).
  3. Reject if postage < MinPostageWei.
  4. Reject if count + pending >= capacity_limit (capacity enforced at
     SEND time so delivery never has to revert — see NIP-0004 §18).
  5. Transfer postage from caller to mailbox owner via SubBalance/
     AddBalance (uint256.Int per StateDB interface). Postage encoded
     in input bytes, NOT msg.value, to avoid CALL value double-debit.
  6. Encode MailboxSendEffect{MailboxID, Sender, PayloadHash,
     SourceBlock}.
  7. mb_pending++, then DqEnqueueDirectly with EffectTypeMailboxSend.
     If enqueue fails (per-block cap or RLP error), mb_pending is
     rolled back so it never sticks high.
  8. Update mb_last_touched_block on the PO.

Block N+1 Phase 0 prologue:
  ProcessDeferredEffects → dispatchDeferredEffect →
  vm.HandleMailboxSendEffect:
    - Decode payload; if RLP corrupt return error (unreachable on
      canonical paths; dispatcher logs and continues without
      reverting the block).
    - Re-validate target mailbox (still exists, not expired, capacity
      not raced). Any "expected" failure → silent drop, return nil.
      mb_pending is released regardless of outcome.
    - Append message slot, advance mb_tail, mb_count++, decrement
      mb_pending.

DqEnqueueDirectly is a NEW exported helper in the Phase 2 deferred
queue that mirrors enqueueEffect byte-for-byte. Both the precompile
selector path and the Mailbox in-process path go through it so per-
block backpressure, monotonic seq minting, deterministic SourceTxHash
derivation, and EIP-161 anchoring live in exactly one place. Any
divergence between the two callers would be a consensus split.

== Constants ==

  MailboxAbsoluteCapacity   = 10000      (hard cap on capacity_limit;
                                          ~1.2 MB max per mailbox)
  MailboxMaxACLEntries      = 64
  MailboxMaxRetentionBlocks = 10_000_000
  MailboxMaxRecvPerCall     = 32         (bound per-tx work)

== Fork gating ==

MailboxForkBlock added to params/ethernova/forks.go, set to 0 on
devnet (active from genesis, consistent with the rest of the NIP-0004
stack: NovenForkBlock, ProtocolObjectForkBlock, DeferredExecForkBlock,
ContentRefForkBlock all 0 on devnet).

Both 0x2C and 0x35 precompiles register at genesis but revert all
selectors pre-fork so there is no incidental state touch on early
blocks. The deferred handler in HandleMailboxSendEffect also gates
on MailboxForkBlock — entries enqueued before the fork drain to
nothing (matches Phase 2 behavior).

Gate is applied identically in the validator path (state_processor.go)
and miner path (worker.go) — asymmetry between these two has been the
canonical consensus-split mode on this chain (cf. SafeTuner postmortem).

== readOnly / EIP-214 enforcement ==

WRITE selectors return ErrWriteProtection in static-call context:
  0x2C: 0x01 createMailbox, 0x02 configureMailbox, 0x03 destroyMailbox
  0x35: 0x01 sendMessage,    0x02 recvMessage

READ selectors are static-call safe:
  0x2C: 0x04 getMailboxConfig
  0x35: 0x03 peekMessage, 0x04 countMessages

== RPC endpoints (ethernova namespace) ==

  ethernova_getMailbox(idHex)                           → MailboxResult | null
  ethernova_getMailboxByOwner(ownerHex, offset, limit)  → list wrapper
  ethernova_getMessages(idHex, fromIndex, limit)        → list wrapper
  ethernova_mailboxConfig()                             → spec metadata
  ethernova_mailboxStats(idHex)                         → queue counters

Precompiles() slice updated to also surface 0x2B (Phase 3 — was
missing from the listing) alongside 0x2C and 0x35 for explorer
clients.

== Determinism / consensus invariants ==

* No Go map iteration anywhere in consensus paths.
* All storage keys are fixed-width 8-byte big-endian derived; no
  float math, no time-of-day reads, no locale-dependent encoding.
* All RLP via rlp.EncodeToBytes (canonical, deterministic).
* Mailbox ID minted from the shared poKeyGlobalNonce monotonic
  counter — guaranteed unique across PO types, no collision with
  Phase 3 ContentRef IDs.
* destroyMailbox does NOT iterate the queue (would be unbounded gas).
  Orphan slots are GC'd by the state expiry sweeper. Mailbox ID is
  unreachable post-destroy because IDs never reuse the global nonce.
* HandleMailboxSendEffect returns nil for all expected failure modes
  (mailbox missing, expired, capacity raced) so block validity is
  never affected by deferred delivery outcomes (NIP-0004 §18).

== What this commit does NOT include ==

* Solidity test harness — to be added after manual compile verification.
* Opcodes MSEND/MRECV/MPEEK/MCOUNT (draft 0xf6–0xf9) — deferred to
  Phase 12 per NIP-0004 §3.4 amendment.
* TTL retention enforcement — config is stored, enforcement reserved
  for a follow-up phase. RetentionPolicy=Never (0) is functionally
  complete; RetentionPolicy=TTL (1) round-trips correctly but does
  not yet sweep expired messages.
* NIP-0003 chat layer on top of Mailbox — separate spec, separate
  phase.
