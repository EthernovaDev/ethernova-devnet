feat(node): make bare ethernova.exe production-ready by default

Change node.DefaultConfig so a no-flag invocation of ethernova.exe
immediately exposes the namespace set required by tooling (hardhat,
ethers, nova-sdk, Phase 8 test suite) without losing the production
safety upstream go-ethereum aims for.

Changes:
- HTTPHost defaults to "localhost" (was ""), so HTTP server starts
  on 127.0.0.1:8545 by default instead of being off.
- WSHost defaults to "localhost" (was ""), so WS server starts on
  127.0.0.1:8546 by default.
- HTTPModules and WSModules default to {eth, net, web3, txpool,
  miner, nova, ethernova} (was {net, web3}). The nova/ethernova
  pair is needed by the Phase 8 RPC surface; eth/txpool/miner cover
  the rest of the standard EVM tooling surface.

Sensitive namespaces (admin, personal, debug) are intentionally kept
opt-in. Operators who need them must add via --http.api admin,debug.
HTTPVirtualHosts stays at {"localhost"}, preserving DNS-rebinding
protection. External exposure still requires the explicit
--http.addr 0.0.0.0 override (same gesture as upstream).

Behavior with CLI flags is unchanged: --http.api still REPLACES the
default list (upstream semantics), so existing scripts that pass an
explicit --http.api are unaffected.

Motivation: the Phase 8 test suite (and any consumer of the nova
SDK) expects nova_*/ethernova_* to be reachable on bare invocations.
Previously they returned -32601 because the HTTP whitelist
defaulted to {net, web3} and required an explicit --http.api flag
to expose anything else.
